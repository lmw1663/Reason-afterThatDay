## 🔴 출시 차단급 결함 (S-tier)

### 1. 위기 핫라인이 검증되지 않은 상태로 코드에 들어 있음
`resources/crisis-hotlines.json:2-4`

```json
"_comment": "출시 전 보건복지부 공식 자료로 반드시 재검증. 잘못된 번호는 생명에 직결되는 위험."
"verified_by": "TODO: 운영자명 입력"
```

앱 자신이 "검증 안 된 채로 쓰지 마"라고 적어 놓고 그 상태로 빌드에 포함돼 있음. `EmotionalCheckModal`의 consecutive_low 1차 CTA가 강제로 이 화면으로 보내기 때문에, 사용자는 검증 안 된 번호를 신뢰하게 됨. **이 한 줄이 출시 가부를 결정함.**

### 2. 청소년 전화(1388)를 24세 이상에게도 노출
6개 핫라인이 평면 리스트로 나열됨. "24세 이하"라는 제약이 description 필드 안에 묻혀 있어, 30대 사용자가 새벽에 스크롤하다 첫눈에 들어오는 번호로 전화할 가능성. 1366(여성 폭력 피해)도 마찬가지로 맥락이 안 맞는 사용자에게 그대로 노출됨. 연령·상황 자기선택 필터 부재.

### 3. consecutive_low 모달이 "지금은 괜찮아"를 눌러도 매번 다시 뜸
`app/(tabs)/index.tsx:63-65`에서 entries 로드될 때마다 `checkConsecutiveLowTemperature` 실행. 닫기를 눌러도 "이 트리거에 대해 24시간 dismiss" 같은 쿨다운이 코드에 없음. **3일 저온인 사용자가 홈에 다섯 번 들어오면 모달도 다섯 번 뜸.** 위로가 아니라 감시이고, 학습된 무력감을 강화함.

## 🟠 사용자 신뢰를 깎는 마찰 (A-tier)

### 4. "재연결 가능성" 메터 자체가 위험한 UX
`app/(tabs)/analysis.tsx`의 3-메터(재연결 / 극복 / 회복)는 disclaimer가 있어도 **숫자처럼 읽히는 시각 요소**임. 아침에 60% → 저녁에 40%를 보면 사용자는 "내가 또 후퇴했다"고 자책함. 이별 직후 사용자에게 "재연결 가능성"을 정량으로 보여주는 것은 미련을 데이터로 합리화시킴 — CLAUDE.md의 "단정 금지" 원칙과 본질적으로 충돌함. **메터 라벨링을 "감정 상태 스냅샷"류로 바꾸거나, 재연결 메터는 제거해야 함.**

### 5. 미니 모드는 AI 응답이 없음 — 무기력한 사용자에게 더 외로운 결과
`app/journal/mini.tsx`는 미니 저장 후 완료 화면만 보여주고 `fetchAiResponse`를 부르지 않음. 의도("가벼운 기록")는 좋지만, 결과적으로:
- 깊게 쓴 날 → 따뜻한 위로
- 무기력해서 슬라이더만 만진 날 → 침묵

**가장 도움이 필요한 날이 가장 빈약한 응답을 받음.** 미니 모드에도 1문장짜리 짧은 응답(템플릿이라도)이 필요함.

### 6. 홈이 매 진입마다 두 가지 안전 체크를 동시 실행
새벽 진입 감지(line 28) + 3일 저온 감지(line 63-65)가 useEffect로 자동 발사됨. 한 번에 두 모달이 큐잉될 수 있고(state는 `safetyAlert` 한 슬롯), 어느 쪽이 이기든 사용자는 **로그인 직후 모달부터 마주함.** "그냥 D+N만 보고 싶었다"는 기본 동기를 차단함. 적어도 첫 진입 후 5초 지연, 또는 사용자가 의도적으로 행동(스크롤 등)한 뒤에 띄워야 함.

### 7. late_night 모달이 "호흡하기"만 제시함
새벽 2시에 앱을 켠 사람의 욕구가 단일하지 않음 — 누군가는 호흡, 누군가는 글쓰기, 누군가는 핫라인. 그런데 1차 CTA는 무조건 "호흡하기"이고 누르면 `/(tabs)` 홈으로 replace 이동함(`EmotionalCheckModal.tsx:40`). **호흡 화면도 아니고 그냥 홈으로 보내는데 라벨은 "호흡하기"** — 약속과 행동이 어긋남. 사용자가 "속았다"고 느낄 첫 순간.

### 8. 일기 내 애정도 슬라이더가 방향 선택 다음에 나타남
`app/journal/today.tsx`의 흐름: 감정 온도 → 라벨 → 신체 → **방향(잡기/보내기/미정)** → 애정도. 실제 인간 인지는 애정의 강도를 먼저 느끼고 그게 방향에 영향을 줌. 순서가 거꾸로면 사용자는 "방향을 정해야 애정도를 입력할 수 있다"는 압박을 받고, 미정을 고를 동기가 줄어듦. 양가감정 진단(최근 커밋 `3c5e4b2`)을 무력화하는 UX 흐름.

## 🟡 누적 사용 시 균열을 만드는 디테일 (B-tier)

### 9. 다크 일색 팔레트(#0E0E12)가 일일 노출 자산으로는 무거움
이별 직후 1주는 잘 맞음. 그러나 D+30 이후 회복기에 진입한 사용자가 매일 같은 거의-검정 배경을 보면 앱 자체가 "슬픈 시기의 잔상"이 됨. **회복 진척도(예: 평균 mood, D+N)에 따라 배경이 미세하게 밝아지는** 동적 톤이 없으면 졸업해도 앱과의 정서 결합이 끊어지지 않음.

### 10. 졸업 유예 Day 1~7이 콘텐츠 고정
`app/cooling/`의 Day별 활동 순서(호흡 → 비교 → 분노 → 차트 → 의미 → 미래 → 확인)가 임의 강제됨. 슬프고 무기력한 사용자에게 "오늘은 미래를 그려봐"는 폭력적일 수 있음. 적어도 "오늘은 다른 Day로 가고 싶어" 옵션은 있어야 함.

### 11. 침습 기억 모달의 마인드풀니스 문장이 단정적
`IntrusiveMemoryModal` Step 2: "그 기억은 사실이지만, 지금의 너는 그 기억 속이 아니야." — **모든 기억이 사실이라는 전제는 틀릴 수 있음** (왜곡·미화·억압). CLAUDE.md의 "단정 금지" 원칙을 위반. "그 기억이 지금 떠올랐어. 지금의 너는 여기에 있어." 정도가 더 안전함.

### 12. 푸시 알림에 badge가 꺼져 있음
`hooks/usePushNotifications.ts:15` `shouldShowBadge: false`. Day 7 최종 푸시처럼 **놓치면 안 되는 1회성 알림**이 있는 앱에서 배지를 끄는 건 의도와 충돌. 사용자가 알림을 그대로 흘려보내면 졸업 절차가 정지됨.

### 13. AI fallback이 방향에 대해 살짝 단정적
`api/ai.ts`의 let_go fallback "보내고 싶은 마음도 용기야" — 고민 중인 사용자가 미정을 고른 뒤 마음이 흔들려서 다시 본 fallback이 "용기야"라고 말하면 결정을 정당화하는 압력으로 작동함. "용기일 수도 있어" 정도의 후행 가설이 system prompt 톤과 일치함.

### 14. 온보딩 3단계에서 감정 라벨 다중선택을 시작 직후에 요구
이별 직후 사용자는 자기 감정에 라벨을 못 붙이는 단계임. 12개 라벨 중 골라야 진행이 자연스러워 보이는 UX는 "내 감정도 정리 못 하는데 이 앱조차 못 쓰겠다"는 첫 패배감을 만들 수 있음. "건너뛰기"가 있다 해도 시각 비중이 약해 발견되기 어려움.
